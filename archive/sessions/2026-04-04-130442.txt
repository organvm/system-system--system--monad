⏺ Ideal Form of Object-Names

  §1. Distinction: Label, Name, Ideal Name

  Three strata of reference, increasing in ontological load:

  Stratum: Label
  Function: Arbitrary tag; extensional pointer
  Example: x, obj_47, UUID
  Failure mode: Opaque — reveals nothing about what is named
  ────────────────────────────────────────
  Stratum: Name
  Function: Conventional sign; partial description
  Example: UserAccount, red_widget
  Failure mode: Leaks implementation or accidents; conflates essence with
  context
  ────────────────────────────────────────
  Stratum: Ideal Name
  Function: Canonical compressed essence; intensional address
  Example: The object's universal property in minimal notation
  Failure mode: —

  A label identifies. A name describes. An ideal name classifies: it positions
  the object in ontological space while individuating it, such that the name is
  recoverable from the object's essence alone.

  ---
  §2. The Naming Problem: Five Constraints

  Let O be a domain of objects, N a set of well-formed names, and ν : O → N a
  naming function.

  Every naming function faces five simultaneous demands:

  ┌──────────────────┬──────────────────────────────────┬───────────────────┐
  │    Constraint    │        Formal requirement        │ What it prevents  │
  ├──────────────────┼──────────────────────────────────┼───────────────────┤
  │                  │ ν is injective: ν(a) = ν(b) ⟹ a  │ Ambiguity (two    │
  │ Individuation    │ = b                              │ objects, one      │
  │                  │                                  │ name)             │
  ├──────────────────┼──────────────────────────────────┼───────────────────┤
  │ Transparency     │ The essence ε(o) is recoverable  │ Opacity (name     │
  │                  │ from ν(o)                        │ hides nature)     │
  ├──────────────────┼──────────────────────────────────┼───────────────────┤
  │                  │ If o = f(o₁,…,oₙ), then ν(o) =   │ Atomicity         │
  │ Compositionality │ f̂(ν(o₁),…,ν(oₙ)) for some        │ (compound objects │
  │                  │ structure-preserving combinator  │  get monolithic   │
  │                  │ f̂                                │ names)            │
  ├──────────────────┼──────────────────────────────────┼───────────────────┤
  │                  │ For each o, ν(o) is unique up to │ Synonymy (one     │
  │ Canonicity       │  equivalence                     │ object, many      │
  │                  │                                  │ names)            │
  ├──────────────────┼──────────────────────────────────┼───────────────────┤
  │                  │ |ν(o)| is minimal subject to the │ Verbosity (name   │
  │ Economy          │  above                           │ exceeds its       │
  │                  │                                  │ referential load) │
  └──────────────────┴──────────────────────────────────┴───────────────────┘

  Theorem (informal). No naming system satisfies all five perfectly in an open
  domain. Ideal naming is therefore a regulative ideal — the form toward which
  naming tends as ontological understanding deepens.

  ---
  §3. Formal Definition

  ★ Insight ─────────────────────────────────────
  The core move: an ideal name is not a string chosen about an object — it is
  the object's self-characterization compressed into a sign. This dissolves the
  Cratylus problem (natural vs. conventional names) by grounding the name in
  structure rather than resemblance or stipulation.
  ─────────────────────────────────────────────────

  Definition. Let (C, Hom) be a category whose objects are the entities to be
  named. An ideal name for an object c ∈ C is a presentation of its universal 
  property — the terminal characterization that determines c uniquely up to
  unique isomorphism.

  Formally, ν*(c) encodes the diagram D and the universal cone (or cocone) such
  that:

  ∀ x ∈ C,  Hom(c, x) ≅ { morphisms satisfying the universal condition }

  Why universal properties? Because:

  1. Individuation — universal properties determine objects up to unique
  isomorphism (the Yoneda lemma guarantees this)
  2. Transparency — the characterization is the essence: what the object does
  for every other object
  3. Compositionality — limits/colimits compose; the name of a product is the
  pair of names
  4. Canonicity — universal properties are unique up to unique isomorphism;
  there is exactly one ideal name per equivalence class
  5. Economy — a universal property is the most compressed characterization: it
  says nothing about internal construction, only about the object's position in
  the relational web

  ---
  §4. The Three-Part Structure

  An ideal object-name decomposes into:

  ν*(o)  =  ⟨ genus · differentia · signature ⟩

  ┌──────────────┬────────────────────────────────────────┬────────────────┐
  │  Component   │                  Role                  │    Recovers    │
  ├──────────────┼────────────────────────────────────────┼────────────────┤
  │ Genus (γ)    │ The kind the object belongs to — its   │ Where it sits  │
  │              │ position in the type hierarchy         │                │
  ├──────────────┼────────────────────────────────────────┼────────────────┤
  │ Differentia  │ What distinguishes it from siblings of │ What makes it  │
  │ (δ)          │  the same genus                        │ this one       │
  ├──────────────┼────────────────────────────────────────┼────────────────┤
  │ Signature    │ Its relational interface — how it      │ What it does   │
  │ (σ)          │ connects to other objects              │                │
  └──────────────┴────────────────────────────────────────┴────────────────┘

  This recapitulates the Aristotelian formula (genus + differentia =
  definition), extended with a relational/categorical component.

  Example in practice:

  Object: The integers
  Genus: Ring
  Differentia: Initial
  Signature: Z : Ring → Set (forgetful)
  Ideal Name: ring--initial or simply ℤ
  ────────────────────────────────────────
  Object: A user's session
  Genus: Temporal-context
  Differentia: Authenticated + bounded
  Signature: (user, duration, permissions)
  Ideal Name: session--authed-bounded
  ────────────────────────────────────────
  Object: A cache eviction policy
  Genus: Eviction-strategy
  Differentia: Least-recently-used
  Signature: (cache, key, time) → evict?
  Ideal Name: eviction--lru

  ★ Insight ─────────────────────────────────────
  The double-separator convention (e.g., --) is not arbitrary: it marks the
  genus–differentia boundary, the single most important joint in any name.
  Everything before -- classifies; everything after -- individuates. This is why
   flat names (getUserData) degrade — they collapse the joint.
  ─────────────────────────────────────────────────

  ---
  §5. Axioms of Ideal Naming

  A1 (Essence over accident). The name encodes necessary properties; contingent
  properties are excluded. A file's name does not encode its creation date
  unless temporality is essential to its kind.

  A2 (Structure over implementation). The name describes what the object is, not
   how it is built. ring--initial not
  set-of-integers-with-addition-and-multiplication.

  A3 (Relation over isolation). When two names share a genus, this is visible in
   their form. Names form a system, not a bag of labels.

  A4 (Compression to canonical form). Among all descriptions satisfying A1–A3,
  the ideal name is the shortest. Redundancy in a name signals that the
  ontological analysis is incomplete — something claimed as differentia is
  already entailed by the genus.

  A5 (Reversibility). Given ν*(o), one can reconstruct the object's essential
  specification. The name is a lossless compression of essence.

  ---
  §6. Consequences

  1. Renaming is reclassification. To change an ideal name is to assert that the
   object's essence, genus, or relational position has changed. Cosmetic
  renaming is incoherent at the ideal level.
  2. Naming difficulty signals ontological confusion. When an object resists
  clean naming, the object's boundaries or kind-membership are unclear. The
  naming problem is the ontology problem.
  3. Synonymy indicates redundant ontology. Two names for one object means the
  domain model contains a distinction without a difference. Merge or refactor.
  4. Hierarchy is intrinsic to naming. Flat namespaces (UUIDs, sequential IDs)
  are maximally non-ideal: pure individuation with zero transparency. Ideal
  names carry their own taxonomy.
  5. The ideal name is the shortest proof of identity. It answers: what is this 
  object, minimally, such that it could be nothing else?

  ---
  §7. Limit Case

  The perfectly ideal name converges with the object itself — the name is the
  universal property, which is the object (up to isomorphism). At this limit,
  naming and defining are the same act. This is the Yoneda embedding: every
  object is fully characterized by how it relates to all other objects, and this
   characterization is its ideal name in the presheaf category.

  yo : C → [Cᵒᵖ, Set]
  yo(c) = Hom(—, c)

  The functor yo(c) is the ideal name of c: complete, canonical, compositional,
  and — via Yoneda — faithful.

  ★ Insight ─────────────────────────────────────
  The Yoneda lemma is, at bottom, a theorem about naming. It says: an object is
  completely determined by the totality of its relationships. Therefore the
  ideal name is not a tag attached to the object but the object's relational 
  fingerprint — the pattern of all morphisms into it. Naming well is just
  ontology done carefully.
  ─────────────────────────────────────────────────

  §4. Design Properties

  ★ Insight ─────────────────────────────────────
  Declaration ≠ instantiation. The env var is cheap (a name in ambient context —
   costs nothing until resolved). The manifest is the specification (carries the
   full ontological weight). Resolution is the act that bridges them — turning a
   declared name into a hydrated instance. This three-phase split mirrors the
  distinction between potentiality, form, and actuality in the Aristotelian
  schema.

  The environment IS the ontological registry. Every P__* variable is a claim
  that a piece exists in this system. piece --env enumerates the ontology. piece
   --validate checks whether declarations have backing specifications. An env
  var without a manifest entry is an unfounded claim — a name without essence.

  Layer numbers encode the contingency cone. Layers 0–2 (distinction, token,
  relation) are universal — their instance.state is "active" in the manifest.
  Layers 3–4 are conditional ("active" but scope-dependent). Layers 5–6 are
  architectural ("dormant" — they exist in specification but require
  activation). The state field tracks where each piece sits on the
  potentiality→actuality axis.
